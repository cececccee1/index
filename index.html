Smart Strategy Map・A物流公司（綜合型3PL）
任務目標：為A物流公司（綜合型3PL：貨運×冷鏈）畫一張A4的「智慧戰略圖」，展開「顧客→指標→手段」三層邏輯。

① 顧客側・VPC 畫布

顧客類型客戶A・7-11零售連鎖客戶B・momo電商客戶C・水產生鮮B2B備註
Jobs（想完成的工作）每日清晨4點，鮮食必到店24h內把訂單送到消費者維持冷鏈不斷鏈今日到貨、帳目清楚、冷鏈不斷
Pains（困擾/痛點）早到客戶來不及，晚到斷貨高峰期延誤、退貨成本溫控偶發異常→整批報廢延遲罰款、溫控異常賠償
Gains（期待的收穫）零違約、可預測全鏈可視區塊鏈級可追溯節省成本、提升準時率、品牌信任

任務目標：延續任務01的A物流公司情境，從「資料」推到「決策」，完成一頁《資料→問題→決策》反思。

Step 1｜掃資料

檔案欄位數/筆數明顯異常觀察

Step 2c｜與講師版本對照

對照題你的觀察
哪裡一致兩個版本的核心內容其實是一樣的：①冷鏈異常是否影響配送效率；②哪些區域/溫層成本高且異常多
哪裡不同1. 管理問題：講師版有說明背景，內容較完整；自己版問題簡潔，一句話就能看懂。2. 用到欄位：自己版有分訂單檔、冷鏈檔
哪個比較好我選擇這三個管理問題，是因為它們同時涵蓋服務品質、成本控制與風險預防三個核心管理面向，且都需要結合貨運與冷鏈資料分析，比單一業務線的問題更貼近綜合型3PL的實際營運需求

Step 3｜寫心得（資料→問題→決策）

資料（Data）問題（Question）決策（Decision）
蒐集並整理訂單配送（配送時間、配送區域、溫層、數量等）與冷鏈溫控資料 從資料中找出管理問題，例如：①冷鏈溫控異常是否影響配送效率 根據分析結果，制定改善措施，例如：優化配送路線、加強溫控監測

我的判斷：部分同意 AI 的分析——AI能快速從訂單配送與冷鏈溫控資料中找出值得關注的管理問題， 提供分析方向；但目前資料缺少訂單與冷鏈之間的共同關聯欄位，且部分資料存在品質問題，仍需人工判斷驗證後才能做出最終決策。

字完整字算法物流意義

問題我的答案

3.2 5等分＋RFM_score

rfm["R_score"] = pd.qcut(rfm["R"], 5, labels=[5, 4, 3, 2, 1]).astype(int)
rfm["F_score"] = pd.qcut(rfm["F"].rank(method="first"), 5, labels=[1, 2, 3, 4, 5]).astype(int)

Section 6｜反直覺收口三問（5 min）

問題我的回答
如果今天的倉是「自動化AGV倉」，你的ABC分類還重要嗎？為什麼？ 重要，但角色會改變。在AGV（自動導引車）倉庫中，商品搬運主要由AGV完成，人員走動已不是主要成本，因此ABC分類不再只是縮短揀貨距離，而是用來優化AGV的行駛路線、補貨順序、儲位利用率等。
如果今天賣的是「季節性服飾」，你的ABC分類有什麼陷阱？ ABC分類會受到時間影響，不能長期固定。例如冬季外套在冬天可能是A類商品，但到了夏天可能變成C類。如果一直沿用舊的ABC分類，可能會讓熱門商品放錯位置，降低揀貨效率。因此季節性商品需要定期重新分析與調整分類。
如果老闆說「下週要降30%倉儲面積」，你會先動A/B/C哪一類？為什麼？ 我會先調整C類商品。因為C類商品出貨量最低、周轉率較低，較適合集中存放、提高儲位密度，甚至評估移至備用倉或降低庫存。A類商品仍應保留在黃金儲位，以維持揀貨效率與服務品質，避免影響出貨速度。

2.3 Pandas Merge四式—商業意義

寫法商業意義（用一句話）
inner只留A與B都有的資料（交集），用於嚴格分析，會漏看單邊獨有的異常
left保留A表全部，B對不到的填NaN，是「以A為主視角」的分析（例如看訂了但還沒到貨的採購單）
right保留B表全部，鏡像left，業界較少用（通常改成調換左右表位置、用left達成同樣效果）
outer保留兩表全部資料，沒對到的都留著填NaN，用來看「哪裡缺漏」，適合稽核用途

2.4 反直覺三點（連連看配對答案）

配對說法
B真正成本：安全庫存×倉儲成本×資金利率
A看到C類庫存上升先看採購紀錄
C你選哪個how反映你選哪個視角

原題：A. 庫存積壓很少是賣不好，常常是買得太兇　B. 供應商「便宜」常常用LT換來　C. Merge不是技術題，是商業假設題

Section 3｜三表Merge實作（15 min）

3.1 Step 1：採購↔進貨（對LT）

import pandas as pd
p = pd.read_csv("purchase.csv", encoding="utf-8-sig", parse_dates=["下單日", "預計到貨"])
r = pd.read_csv("receipt.csv",  encoding="utf-8-sig", parse_dates=["實際到貨"])
# 你選的how = "left"
pr = p.merge(r, on=["採購單號", "SKU"], how="left")
# LT三層
pr["實際LT"]   = (pr["實際到貨"] - pr["下單日"]).dt.days
pr["計畫LT"]   = (pr["預計到貨"] - pr["下單日"]).dt.days
pr["LT延遲日"] = pr["實際LT"] - pr["計畫LT"]
print(pr.head())

你選的how＝"left"
理由：以採購單為主視角，保留「訂了但還沒到貨」的採購單，才能看出斷貨追責的早期警訊

✅ 觀察：Merge後筆數＝102（原p＝102筆／原r＝73筆）

3.2 Step 2：加銷售

s = pd.read_csv("sales.csv", encoding="utf-8-sig", parse_dates=["出貨日"])
sku_sales = s.groupby("SKU")["出貨量"].sum().reset_index()
sku_sales.columns = ["SKU", "九月銷量"]
full = pr.merge(sku_sales, on="SKU", how="left")
full["九月銷量"] = full["九月銷量"].fillna(0)
full["庫存積壓量"] = full["實際數量"] - full["九月銷量"]
